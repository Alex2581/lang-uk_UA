; page intro text
:parityProblems_intro_plug:
**Помічник з проблем парності** є частиною плагіна **Налаштування перевірки парності**.<p/>
Поточний статус – **ЕКСПЕРИМЕНТАЛЬНИЙ**, і завжди можливо, що цього помічника буде видалено в майбутньому). 
**(вітаються відгуки про основну концепцію та ідеї щодо вдосконалення)**.<p/>
Ідея полягає в тому, що цей помічник можна використовувати після того, як перевірка парності або перевірка читання повідомили про проблеми та занесли деталі будь-яких проблемних секторів до системного журналу. 
У багатьох випадках такі помилки можуть бути викликані зовнішніми факторами, такими як кабелі чи джерело живлення, а не фактичними дисками.<p/>
Також нерідко, особливо в системах на базі **Ryzen/ThreadRipper**, випадкові помилки парності, якщо оперативна пам’ять розігнана.
(профіль XMP **є** розгоном), 
і комбінація материнська плата/ЦП також може накладати обмеження, нижчі за оперативну пам’ять, яку здатна підтримувати.<p/>
Якщо ви вважаєте, що усунули те, що спричинило проблеми, ви можете скористатися цим помічником, щоб запустити часткову перевірку у вузькому діапазоні, щоб побачити, чи проблема все ще виникає.  
Це може бути набагато швидше, ніж виконувати повну перевірку парності чи перевірку читання, щоб побачити, чи ви вирішили проблему, або запускати розширений тест SMART диска, щоб перевірити фактичний диск.<p/>
**ВАЖЛИВО**: цей помічник ніколи не повинен використовуватися як альтернатива запуску повної перевірки парності, якщо ви вважаєте, що проблему вирішено.
:end

:parity_tune_explain_plug:
> Основна мета цього плагіна — дозволити вам обмежити операції з масивами, такі як перевірка парності, виконанням у часи, які не завдаватимуть вам незручностей.
>
>Якщо ви встановили для цього значення **Ні**, тоді ви отримаєте поведінку системи за замовчуванням, коли перевірки парності виконуються без перерви до завершення, якщо ви вручну не зупините/призупините/скасуєте їх.
>
>Встановлення для цього параметра значення **Так** вказує на те, що перевірки парності мають виконуватися поетапно протягом кількох днів.
> Припустимо, як приклад того, що може зробити цей плагін
>У вас є перевірка паритету, яку ви запланували починати першого числа кожного місяця опівночі  
>* Ваш минулий досвід показав, що якщо перевірка парності виконується безперервно, вона займає 30 годин.  
>* Ви встановлюєте цей плагін на використання кроків у 3 години, починаючи з опівночі й закінчуючи о 3:00 ранку.  
>* Перевірка парності тепер фактично триватиме 10 днів (10 x 3 = 30), тому перевірка парності завершиться 10 числа місяця.  
>* Ви запланували виконання цих приростів, починаючи з опівночі й закінчуючи о 3:00 ночі. коли ви знаєте, що система не використовується.  
>* Вас не хвилює цей збільшений час, що минув, і ви будете раді підвищеній чуйності системи під час звичайного використання.  
>* Решту місяця цей плагін нічого не робитиме, оскільки не виконується активна перевірка парності, коли настане час початку збільшення.
>
>**ОБЕРЕЖНО:** Якщо масив зупинено, коли операція масиву не завершена, поточний прогрес буде втрачено, і його можна буде перезапустити лише з початку
>* Перевірка парності тепер фактично триватиме 10 днів (10 x 3 = 30), тому перевірка парності завершиться 10 числа місяця.  
:end

:parity_tune_hot_plug:
> Pause an array operation(Parity Check, Parity-Sync/Disk Rebuild, disk Clear)if the disk temperatures exceeds the limits you have set. 
>
>The temperatures are checked against the thresholds set for the Warning disk temperature levels. If a threshold has been defined for an individual drive (accessed by clicking on the drive in the Main tab) then this is the value used.  If not the global setting (set via *Settings->Display Settings*) will be used.  It is assumed that normally you would want the array operation to be paused before the value set at the unRaid level as the Warning Threshold.
>
>A much better solution is to improve the cooling in your case so that the disks never overheat.   In practice this may not always prove practical.
>
>If the array operation was part of running an increment then it will not be resumed outside the time allotted for the increment.   If the array operation was initiated for any other reason then the Pause/Resume behavior on temperature is always active.

:parity_tune_warn_plug:

If an explicit threshold has been defined for an individual drive then this is the value used.  If not the global setting will be used.
>
> You normally want a small positive value to trigger the pause before Unraid gets around to sending you a notification that the temperature warning threshold has been reached for a disk.
>
> As an example if the Warning threshold for a drive is 50C then setting a pause value of 5 would cause the plugin to pause the operation if that drive reached 45C (50-5).
>
>If there is no active array operation then no action will be taken even if disk temperatures exceed the specified threshold.
>
> If there is no running array operation then no action will be taken.
:end

>
>If a disk ever gets spun down the temperature is not readily available so it will be assumed that this criteria has been met
>
