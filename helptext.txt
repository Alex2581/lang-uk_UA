**Ідентифікація** — це *підпис*, який унікально ідентифікує пристрій зберігання. Підпис
містить номер моделі пристрою, серійний номер, ідентифікатор пристрою Linux і розмір пристрою.

**Темп.** (температура) зчитується безпосередньо з пристрою. Ви налаштовуєте, на яких одиницях використовувати
сторінку [Параметри відображення](Settings/DisplaySettings). Ми не зчитуємо температуру жорсткого обертання
накопичувачі, оскільки це зазвичай спричиняє їх обертання; замість цього ми відображаємо символ `*`. Ми також
відображати символ `*` для SSD і флеш-пристроїв, хоча іноді ці пристрої повідомляють про дійсність
температури, а іноді вони повертають значення `0`.

**Розмір, Використано, Вільно** повідомляє про загальний розмір пристрою, використаний простір і залишок місця для файлів. Ці
одиниці також налаштовуються на сторінці [Параметри відображення](Налаштування/Параметри відображення). The
обсяг використаного простору буде відмінним від нуля навіть для порожнього диска через навантаження на файлову систему.

*Примітка: для кеш-пулу з кількома пристроями ці дані стосуються всього пулу, які повертає btrfs.*

**Читання, запис** – це кількість запитів вводу-виводу, надісланих драйверам вводу-виводу пристрою. Ця статистика може
скинути в будь-який час, зверніться до розділу Статус масиву нижче.

**Помилки** підраховує кількість *невиправних* помилок, про які повідомляє пристрій
Драйвери введення/виведення. Відсутні дані через невиправні помилки читання масиву заповнюються на льоту за допомогою паритету
реконструювати (і ми намагаємося записати ці дані назад у сектор(и), який не вдався). Будь-який невиправний
помилка запису призводить до *вимкнення* диска.

**FS** вказує на файлову систему, виявлену в розділі 1 пристрою.

Стовпець **Перегляд** містить піктограму папки, яка вказує на те, що пристрій *підключено*. Натисніть піктограму, щоб
переглядати файлову систему.

Якщо на сторінці [Параметри відображення] (Налаштування/Параметри відображення) увімкнено параметр «Відображати підсумки масиву»,
Включено рядок **Загальний**, який містить статистичні дані пристрою, включаючи середню температуру
ваших пристроїв.

Масив потрібно зупинити, щоб змінити призначення пристрою масиву.

Масив Unraid складається з одного або двох дисків Parity та кількох дисків Data. Дані
диски використовуються виключно для зберігання даних користувача, а диск(и) Parity забезпечує необхідну надмірність
для відновлення після збою диска.

Оскільки дані не розрізняються по масиву, диск(и) Parity має бути таким же великим або більшим, ніж найбільші дані
диск. Парність також має бути вашою найкращою продуктивністю.

Кожен диск даних має власну файлову систему і може бути експортований як
окрема частка.

Натисніть назву пристрою, щоб налаштувати окремі параметри пристрою та запустити певні утиліти.
:end

**Пул пристроїв** — це один пристрій або пул із кількох пристроїв *за межами* масиву Unraid. Його можна експортувати лише для доступу до мережі

Існує два способи налаштування пристроїв Pool:

1. Як єдиний пристрій, або
2. Як пул із кількома пристроями.

Якщо налаштовано як один пристрій, ви можете відформатувати пристрій за допомогою будь-якої підтримуваної файлової системи (btrfs, reiserfs,

один пристрій пулу виходить з ладу, усі дані, що містяться на ньому, можуть бути втрачені.

Якщо налаштовано як пул кількох пристроїв, Unraid OS автоматично вибере формат *btrfs-raid1* (для обох даних
і метадані). btrfs дозволяє додавати до пулу будь-яку кількість пристроїв, і кожна копія даних гарантується
для запису на два різні пристрої. Таким чином, пул може витримати збій одного диска без втрати даних.

Якщо [User Shares](/Settings/ShareSettings) увімкнено, спільні користувачі можуть бути налаштовані на
автоматично використовувати пристрій Pool, щоб
прискорити запис. Можна запланувати запуск спеціального фонового процесу, який називається *Перемістити*
періодично переміщувати спільні файли користувача з кешу в масив.

#### Призначення пристроїв

Дисковий масив unRAID складається з одного диска парності та кількох дисків із даними. Дані
диски використовуються виключно для зберігання даних користувача, а диск парності забезпечує необхідну надмірність
для відновлення після будь-якої відмови окремого диска.

Зауважте, що ми обережно використовуємо термін *диск*, коли йдеться про накопичувальний пристрій масиву. Ми
використовуемо термін *жорсткий диск* (або іноді просто *диск*), коли йдеться про справжній жорсткий диск (HDD)
пристрій. Це пояснюється тим, що в системі RAID можна читати/записувати дисковий масив, відповідний
жорсткий диск вимкнено або навіть відсутній! Крім того, корисно вміти запитати, «який прилад
призначений як диск парності?"; або, "який пристрій відповідає диску2?".

Тому нам потрібен спосіб призначити жорсткі диски дискам-масивам. Це досягається тут, на
Головній сторінці, коли масив зупинено. Для кожного диска масиву є спадне вікно, у якому перераховано всі
непризначені пристрої. Щоб призначити пристрій, просто виберіть його зі списку. Кожного разу, коли виконується
призначення пристрою, система оновлює файл конфігурації для запису призначення.

#### Вимоги

На відміну від традиційних систем RAID, які розподіляють дані на всіх пристроях масиву, сервер unRAID
зберігає файли на окремих жорстких дисках. Отже, усі операції запису файлу включатимуть обидва
диск даних, на який записується файл, і диск парності. З цих причин,

* розмір диска парності має бути таким же чи більшим, ніж будь-який диск із даними,

і

* за наявності вибору диск парності має бути найшвидшим диском у вашій колекції.

#### Настанови

Нижче наведено кроки, які слід виконувати під час проектування дискового масиву unRAID.

1. Вирішіть, який жорсткий диск ви будете використовувати для паритету, а які жорсткі диски 
ви будете використовувати для даних disk1, disk2 тощо, і позначте їх певним чином. Крім того, знайдіть 
серійний номер кожного жорсткого диска та запишіть його десь; ця інформація вам знадобиться пізніше.

2. Встановіть жорсткі диски, завантажте сервер unRAID і відкрийте веб-інтерфейс. Якщо це нова 
збірка системи, на головній сторінці буде показано відсутність встановлених дисків. Це не означає, 
що система не може виявити ваші жорсткі диски; це просто означає, що жодного ще не було призначено.

3. Пам’ятаєте серійні номери, які ви записали на кроці 1? Для парності та кожного диска даних виберіть
правильний жорсткий диск на основі його серійного номера зі випадаючого списку.

#### Гаряче підключення

Ви також можете *На Гарячу підключити* жорсткі диски до свого сервера, якщо ваше обладнання це підтримує. Наприклад,
якщо ви використовуєте каркаси для жорстких дисків, ви можете просто підключити їх до увімкненого сервера
з Зупиненим масивом. Оновіть головну сторінку, щоб у призначенні з’явилися нові непризначені пристрої
випадаючі списки.

#### Наступні кроки

Після того, як ви призначите всі свої жорсткі диски, зверніться до розділу «Стан масиву» нижче
і Запустіть масив.
:end

#### Введення шифрування

З Зупиненим масивом користувач може вказати новий ключ шифрування. Зверніть увагу, що як тільки пристрій
відформатований за допомогою певного ключа, його можна відкрити лише за допомогою того самого ключа. Зміна ключа шифрування вимагає
зашифровані пристрої, які потрібно переформатувати, **що призведе до остаточної втрати всіх існуючих даних на цих пристроях.**

#### Парольна фраза

Введіть парольну фразу довжиною до 512 символів. Настійно рекомендується використовувати лише 95 друкованих символів із
перших 128 символів [таблиці ASCII](https://en.wikipedia.org/wiki/ASCII), оскільки вони завжди матимуть однаковий двійковий код
представництво. Інші символи можуть мати інше кодування залежно від конфігурації системи, і ваша парольна фраза
не працюватиме з іншим кодуванням. Якщо ви бажаєте довшу парольну фразу або включити двійкові дані, замість цього завантажте файл ключа.

Будь ласка, зверніться до [cryptsetup FAQ](https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#5-security-aspects)
для того, що становить *безпечну* парольну фразу.

**Запам'ятати** цей пароль. **ЯКЩО ВТРАЧЕНО, ЗАШИФРОВАНИЙ ВМІСТ ВІДНОВИТИ НЕМОЖЛИВО!**

#### Ключовий файл

Виберіть локальний ключовий файл із збереженим ключем шифрування або двійковий файл. Максимальний розмір файлу ключів становить 8 Мб (8388608 байт).

**Назад** ваш локальний файл ключів. **ЯКЩО ВТРАЧЕНО, ЗАШИФРОВАНИЙ ВМІСТ ВІДНОВИТИ НЕМОЖЛИВО!**
:end

:info_btrfs_balance_help:
**Balance** запустить програму *btrfs balance*, щоб змінити межі на всіх пристроях пулу, наприклад,
для перетворення пулу з raid1 на raid0 або навпаки.

Коли виконується *повний баланс*, він фактично переписує все в поточній файловій системі.

*Перетворення режиму* впливає на екстенти даних btrfs; метадані завжди використовують raid1 і за потреби перетворюються на raid1 будь-якою операцією балансу.

Час роботи може бути дуже довгим, залежно від розміру файлової системи та швидкості пристрою.

Під час створення пулу з кількома пристроями Unraid використовує ці параметри за замовчуванням:

`-dconvert=raid1 -mconvert=raid1`

Щоб отримати повнішу документацію, зверніться до btrfs-balance [Manpage](https://btrfs.wiki.kernel.org/index.php/Manpage/btrfs-balance)

*Примітка: raid5 і raid6 зазвичай все ще вважаються **експериментальними** спільнотою Linux*
:end

:info_btrfs_check_help:
**Перевірити** запустить програму *btrfs check* для перевірки цілісності файлової системи на пристрої.

Поле *Параметри* ініціалізується за допомогою *--readonly*, що визначає лише перевірку. Якщо потрібен ремонт, 
ви повинні запустити другий проход перевірки, встановивши *Параметри* на *--repair*; це дозволить *btrfs check* виправити файлову систему.

:share_edit_split_level_help:
Визначає, чи дозволено каталогу розгортатися на кілька дисків.

**Автоматично розділяйти будь-який каталог за потреби**
Коли потрібно створити новий файл або підкаталог у спільному ресурсі, Unraid OS спочатку вибирає, на якому диску
його слід створити, відповідно до налаштованого *Методу розподілу*. Якщо батьківський каталог містить
новий файл або підкаталог, не існує на цьому диску, тоді Unraid OS спочатку створить усі необхідні
батьківські каталоги, а потім створіть новий файл або підкаталог.

**Автоматично розділяти лише каталог верхнього рівня за потреби**
Коли створюється новий файл або підкаталог у підкаталозі першого рівня спільного ресурсу, якщо це перший
на диску, який записується, не існує підкаталогу рівня, тоді спочатку буде створено підкаталог. Якщо новий 
файл або підкаталог створюється в підкаталозі другого або нижчого рівня спільного ресурсу, новий файл або підкаталог 
створюється на тому самому диску, що й батьківський каталог нового файлу або підкаталогу.

**За потреби автоматично розділяти лише каталоги верхнього рівня "N"**
Подібно до попереднього: коли створюється новий файл або підкаталог, якщо батьківський каталог знаходиться на рівні "N",
і не існує на вибраному диску, Unraid OS спочатку створить усі необхідні батьківські каталоги.  Якщо

:share_edit_cache_pool_help:
Укажіть, чи можна нові файли та каталоги, записані на спільному ресурсі, записувати на кеш-диск/пул, якщо він є.
Цей параметр також впливає на поведінку *Переміщення*.

**Ні** забороняє запис нових файлів і підкаталогів на кеш-диск/пул.
*Переміщення* не виконуватиме жодних дій, тож усі існуючі файли для цього спільного ресурсу, які знаходяться в кеші, залишатимуться там.

**Так** означає, що всі нові файли та підкаталоги мають бути записані на кеш-диск/пул, якщо
достатньо вільного місця на кеш-диску/пулі.
Якщо на кеш-диску/пулі недостатньо місця, у масиві створюються нові файли та каталоги.
Коли викликається *mover*, файли та підкаталоги переносяться з кеш-диска/пулу в масив.

**Тільки** вказує на те, що всі нові файли та підкаталоги мають бути записані на кеш-диск/пул.
Якщо на кеш-диску/пулі недостатньо вільного місця, операції *створення* завершаться помилкою зі статусом *недостатньо місця*.
*Mover* не виконуватиме жодних дій, тож усі існуючі файли для цього спільного ресурсу, які знаходяться в масиві, залишатимуться там.

**Перевага** вказує, що всі нові файли та підкаталоги мають бути записані на кеш-диск/пул, якщо
достатньо вільного місця на кеш-диску/пулі.
Якщо на кеш-диску/пулі недостатньо місця, у масиві створюються нові файли та каталоги.
Коли викликається *mover*, файли та підкаталоги переносяться з масиву на кеш-диск/пул.

**ПРИМІТКА:** Mover ніколи не переміщуватиме файли, які зараз використовуються.
Це означає, що якщо ви хочете перемістити файли, пов’язані із системними службами, такими як Docker або віртуальні машини, вам це потрібно

:end

:smb_case_sensitive_names_help:
Контролює, чи назви файлів чутливі до регістру.

Налаштування за замовчуванням **Авто** дозволяє клієнтам, які підтримують чутливі до регістру імена файлів (Linux CIFSVFS), повідомляти сервер 
Samba на основі кожного пакета, що вони бажають отримати доступ до файлової системи з урахуванням регістру (для підтримки регістру UNIX чутлива 
семантика). Жодна система Windows не підтримує чутливі до регістру назви файлів, тому встановлення цього параметра на **авто** є таким самим, як 
встановлення значення Ні для них; однак регістр імен файлів, переданих клієнтом Windows, буде збережено. Цей параметр може призвести до 
зниження продуктивності з дуже великими каталогами, оскільки Samba має виконувати пошук імен файлів і збігатися з переданими іменами.

Параметр **Так** означає, що файли створюються з регістром, який передає клієнт, і доступні лише за допомогою цього самого регістру.
Це прискорить доступ до дуже великого каталогу, але деякі програми Windows можуть не працювати належним чином із цим параметром.  
Наприклад, якщо "MyFile" створено, але програма Windows намагається відкрити "MYFILE" (що дозволено в Windows), його не буде знайдено.

Значення **Змусити знизити** є особливим: регістр усіх вхідних імен файлів клієнта, а не лише нових імен файлів, буде встановлено в нижньому регістрі.
Іншими словами, незалежно від того, яке ім’я зі змішаним регістром буде створено на стороні Windows, воно зберігатиметься та використовуватиметься лише в нижньому регістрі.  
Це гарантує, що всі програми Windows правильно знайдуть будь-який файл незалежно від регістру, але регістр не буде збережено в списках папок.
Зауважте, що цей параметр слід налаштовувати лише для нових спільних ресурсів.
:end

налаштування практично для всіх таблиць розділів у стилі MBR.

Параметр **MBR: 4K-aligned** створить таблицю розділів у стилі MBR, де один
розділ 1 почнеться в **64-му секторі** від початку диска. Оскільки розмір сектора становить 512 байт,
це *вирівняє* початок розділу 1 на межі 4 КБ. Це потрібно для правильного
підтримка так званих дисків *Advanced Format*.

Якщо у вас немає особливих вимог, не змінюйте цей параметр із стандартного **MBR: 4K-aligned**.
:end

**Provision** may be used to allocate a *free* SSL Certficiate from [Let's Encrypt](https://letsencrypt.org/) and
then upload to your server. Note: We **highly** recommend using a static IP address in this case.

**Update DNS** may be used to manually initiate updating the DNS A-record of your server FQDN on unraid.net. Note
that DNS propagation change could take anywhere from 1 minute to several hours (we set TTL to 60 seconds).

Note: **Provision** may fail if your router or upstream DNS server has
[DNS rebinding protection](https://en.wikipedia.org/wiki/DNS_rebinding) enabled. DNS rebinding
protection prevents DNS from resolving a private IP network range. DNS rebinding protection is meant as
a security feature on a local LAN which includes legacy devices with buggy/insecure "web" interfaces.

One source of DNS rebinding protection could be your ISP DNS server. In this case the problem may be solved by
switching to a different DNS server such as Google's public DNS.

More commonly, DNS rebinding protection could be enabled in your router. Most consumer routers do not implement DNS
rebinding protection; but, if they do, a configuration setting should be available to turn it off.

Higher end routers usually do enable DNS rebinding protection however. Typically there are ways of turning it off
entirely or selectively based on domain. Examples:

**DD-WRT:** If you are using "dnsmasq" with DNS rebinding protection enabled, you can add this line to your router
configuration file:

`rebind-domain-ok=/unraid.net/`

**pfSense:** If you are using pfSense internal DNS resolver service, you can add these Custom Option lines:

`server:`<br>
`private-domain: "unraid.net"`

**Ubiquiti USG router:** you can add this configuration line:

`set service dns forwarding options rebind-domain-ok=/unraid.net/`

**OpenDNS:** Go to Settings -> Security and *remove* the checkbox next to
 "Suspicious Responses - Block internal IP addresses". It is an all-or-nothing setting.

When all else fails, you can create an entry in your PC's *hosts* file to override external DNS and
directly resolve your servers unraid.net FQDN to its local IP address.

:nfs_tunable_fuse_remember_help:
When NFS is enabled, this Tunable may be used to alleviate or solve instances of "NFS Stale File Handles"
you might encounter with your NFS client.

In essence, (fuse_remember) tells an internal subsystem (named "fuse") how long to "remember" or "cache"
file and directory information associated with user shares.  When an NFS client attempts to access a file
(or directory) on the server, and that file (or directory) name is not cached, then you could encounter
"stale file handle".

The numeric value of this tunable is the number of seconds to cache file/directory name entries,
where the default value of 330 indicates 5 1/2 minutes.  There are two special values you may also set
this to:

* 0 which means, do not cache file/directory names at all, and
* -1 which means cache file/directory names forever (or until array is stopped)

A value of 0 would be appropriate if you are enabling NFS but only plan to use it for disk shares,
not user shares.

:eth_enable_bonding_help:
Bonding is a feature that combines multiple physical Ethernet interfaces into a single *bonded* interface named **bond0**.
This can be used to improve the connection redundancy and/or throughput of the system.
Different bonding modes are supported (see below), but some modes require proper switch support.
:end

:eth_bonding_mode_help:
**Mode 0 (balance-rr)**<br>
This mode transmits packets in a sequential order from the first available slave through the last.
If two real interfaces are slaves in the bond and two packets arrive destined out of the bonded interface the first will be transmitted on the first slave and the second frame will be transmitted on the second slave.
The third packet will be sent on the first and so on. This provides load balancing and fault tolerance.

**Mode 1 (active-backup) - default**<br>
This mode places one of the interfaces into a backup state and will only make it active if the link is lost by the active interface.
Only one slave in the bond is active at an instance of time. A different slave becomes active only when the active slave fails.
This mode provides fault tolerance.

**Mode 2 (balance-xor)**<br>
This mode transmits packets based on an XOR formula. Source MAC address is XOR'd with destination MAC address modula slave count.
This selects the same slave for each destination MAC address and provides load balancing and fault tolerance.

**Mode 3 (broadcast)**<br>
This mode transmits everything on all slave interfaces. This mode is least used (only for specific purpose) and provides only fault tolerance.

**Mode 4 (802.3ad)**<br>
This mode is known as *Dynamic Link Aggregation*. It creates aggregation groups that share the same speed and duplex settings.
It requires a switch that supports IEEE 802.3ad dynamic link.
Slave selection for outgoing traffic is done according to the transmit hash policy, which may be changed from the default simple XOR policy via the xmit_hash_policy option.
Note that not all transmit policies may be 802.3ad compliant, particularly in regards to the packet mis-ordering requirements of section 43.2.4 of the 802.3ad standard.
Different peer implementations will have varying tolerances for noncompliance.

**Mode 5 (balance-tlb)**<br>
This mode is called *Adaptive transmit load balancing*. The outgoing traffic is distributed according to the current load and queue on each slave interface.
Incoming traffic is received by the current slave.

**Mode 6 (balance-alb)**<br>
This mode is called *Adaptive load balancing*. This includes balance-tlb + receive load balancing (rlb) for IPV4 traffic.
The receive load balancing is achieved by ARP negotiation.
The bonding driver intercepts the ARP Replies sent by the server on their way out and overwrites the src hw address with the unique hw address of one of the slaves in the bond
such that different clients use different hw addresses for the server.

*Mode 1 (active-backup) is the recommended setting. Other modes allow you to set up a specific environment, but may require proper switch support.
Choosing a unsupported mode can result in a disrupted communication.*
:end
